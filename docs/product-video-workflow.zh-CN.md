# 产品一致性优先的四视图图生视频工作流

## 目标

本项目只服务一个核心目标：用 AI 生成电商产品视频时，先保证“货对版”，再追求动作、场景和时长。

优先级固定为：

1. 产品一致性
2. 动作有趣
3. 场景丰富
4. 时长更长

任何生成策略、界面流程、prompt 和接口参数都不能反过来压过产品一致性。如果场景或动作会导致产品漂移，必须牺牲场景和动作。

## 适用品类

当前长期预设品类是五个可穿戴充气服：鲨鱼、奶牛、灰色老鼠、青蛙、相扑。

品类级硬规则：

- 产品必须始终是可穿戴充气服，不能变成真实动物、卡通角色、毛绒玩具、CG 生物或重新设计的吉祥物。
- 必须保留充气尼龙/塑料布料、褶皱、接缝、充气体积、拉链或穿戴入口、脚套或鞋子、侧面/背部附加结构、鼓风阀或进气口。
- 不得新增产品图里不存在的嘴、牙齿、眼睛、毛发、爪子、鳞片、logo、文字、配饰或图案。

## 四个用户可见步骤

界面只露出 4 个步骤：

```text
上传四视图 -> 生成首帧 -> 生成视频 -> 视频质检
```

“定款/锁定产品细节”是系统后台自动步骤，不作为用户可点击步骤露出，也不要求用户逐条确认。用户真正需要确认的是生成后的首帧是否货对版。

## 1. 上传四视图

上传阶段必须先提供四张平行的核心视图，没有主图和参考图之分：

1. 正面图：锁定白肚、透明脸窗、拉链、正面比例、脚部。
2. 左侧图：锁定左侧厚度、左侧轮廓、左侧可见的眼睛/鳃线/侧鳍/接缝，以及该侧缺失的结构。
3. 右侧图：锁定右侧厚度、橙色鼓风阀方向和高度、右侧侧鳍、侧缝、尾部边缘和鞋底。
4. 背面图：锁定纯蓝背部、中轴尾鳍、背部竖缝、背面轮廓。

用户不再上传细节图。长期产品可以在本地预设中保存后台辅助角度，用来强化阀门、尾巴、围巾、腰带、拉链、缝线、褶皱或材质等易漂移部位。

上传规则：

- 正面、左侧、右侧、背面四张核心图全部上传后，才允许进入首帧。
- 上传正面图后不能自动跳到首帧。
- 不接受只上传一张图或只上传正面图。
- 不把视频作为必需输入；当前产品一致性策略以四张静态图片为准。
- 前端提交给首帧接口的核心字段是 `image_urls`，且必须包含 4 张可读核心图片。
- `image_urls` 的语义顺序固定：`image_urls[0]` 是正面，`image_urls[1]` 是左侧，`image_urls[2]` 是右侧，`image_urls[3]` 是背面。
- 本地预设辅助角度提交到 `support_image_urls`，由系统自动附带，不作为用户上传入口。
- 后端必须拒绝 `blob:` 本地预览地址，只接受 `data:image/` 或 `http(s)` 图片地址。

## 2. 后台自动定款

四视图上传完成后，系统自动生成产品一致性合同。这个合同不暴露成用户步骤，但必须进入首帧和视频 prompt。

一致性合同包含：

- 品类级锁定：始终是可穿戴充气服。
- 四视图拓扑：正面、左侧、右侧、背面各自约束真实物理位置。
- 易漂移部位：阀门、透明脸窗、拉链、尾鳍、鳃线、鞋子、褶皱、接缝；本地预设辅助角度只作为补充证据强化这些点。
- 禁止变化：不得重绘、平均、拼贴、挪位、复制、删减、放大、缩小或风格化产品。
- 人体体型包络：产品必须像真人穿着一件轻到中度充气服，外壳只比真人肩宽和躯干略宽；不能变成巨大圆顶头、桶状身体、站立气球、展示道具或吉祥物外壳。
- 体积包络：保持四视图共同的尺寸、比例、厚度和中低充气轮廓。

四视图不是拼贴素材，也不是装饰参考。它们是同一个产品的拓扑地图。

## 3. 生成首帧

首帧模型只做一件事：

```text
在指定场景里生成一张产品入景图，同时严格保持四视图定义的产品结构。
```

首帧生成规则：

- 必须基于四张核心 `image_urls` 同时生成，可叠加本地预设 `support_image_urls` 作为同款产品辅助证据。
- 产品一致性高于场景描述，场景只能是低优先级输入。
- 不能把四张图平均成一个新产品。
- 不能把四张图拼成一个不可能的表面。
- 必须先选择一个物理合理的镜头族：正面、左侧、右侧或背面。
- 看不见的细节应该自然隐藏，不能挪到错误位置。
- 画面必须在所选视角下尽量完整展示产品，不裁切该视角物理可见的关键部位。
- 不要把侧面或背面才可见的细节强行塞进正面图。例如正面镜头里侧阀可以隐藏或只在窄侧边微微可见，背部尾鳍在非背面镜头里应该隐藏。

视角可见性矩阵：

- 正面镜头：可见白肚、透明脸窗、拉链，只允许看到很窄侧边。
- 左侧镜头：只可见左侧图里存在的眼睛、鳃线、侧鳍、接缝、厚度等结构。
- 右侧镜头：只可见右侧图里存在的阀门、侧鳍、接缝、厚度、尾部边缘等结构。
- 背面镜头：可见纯蓝背部、中轴尾鳍、背部竖缝。

## 4. 首帧审核

首帧必须人工确认后才能进入视频。

审核清单：

- 产品仍然是同一件可穿戴充气服。
- 人体体型包络、尺寸、比例、轮廓、厚度和充气程度与四视图一致。
- 一旦首帧像巨大吉祥物、桶状气球、全鼓展示道具，或头部/躯干明显比四视图更大，必须判错误并重新生成。
- 透明脸窗、拉链、白肚、左右侧厚度、侧眼、5 条鳃线、橙色阀门方向、尾鳍、黑鞋位置正确。
- 没有多手、多鳍、多尾、错位阀门、错位尾鳍或虚构结构。
- 产品没有被场景遮挡关键细节。
- 产品没有被裁切。

所有 critical 审核项必须逐项判断“正确”或“错误”后，才能批准首帧。任一 critical 项判断为错误时，本次首帧失败，必须重新生成；不能用“未勾选”来代替失败判断。

只要首帧不过审，就不能进入视频。

## 5. 生成视频

视频模型只负责动画化已批准首帧，不负责重新理解产品。

已批准首帧是视频模型的直接视觉输入。四视图继续作为文本合同和元数据锁存在，除非所选视频 API 明确支持额外视觉参考图。默认必须把镜头限制在已批准首帧的视角族内，避免展示未经首帧证明的新表面。

默认高一致性动作预算：

- 旋转 0-8 度。
- 小幅充气弹动。
- 小幅手鳍动作。
- 极小一步位移。
- 不转身、不大范围走动、不跳、不快速跳舞、不切镜头、不展示未经批准的新角度。

视频 prompt 必须继承：

- 已批准首帧。
- 四张核心视图产品硬锁。
- 品类级可穿戴充气服锁定。
- 视角拓扑锁定。
- 体积包络锁定。
- 用户动作描述，且动作描述低于产品一致性。

如果动作与产品一致性冲突，忽略动作。

## 6. 视频质检与重试

视频生成后至少检查 5 个时间点：

- 0%
- 25%
- 50%
- 75%
- 100%

通过标准：

- 货对版评分至少 90。
- 动作趣味目标至少 70。
- 场景丰富目标至少 60。
- 时长只作为信息，不压过货对版。

如果产品失败，重试策略必须收紧：

- 降低旋转。
- 减少动作。
- 简化场景。
- 使用更静态镜头。
- 强化易漂移细节锁定。
- 必要时缩短时长。

如果场景很好但产品漂移，该次生成失败。

## 鲨鱼充气服默认锁定点

正面：

- 白色腹部区域。
- 横向透明脸窗。
- 脸窗下方垂直拉链。
- 蓝色外边。
- 白色内侧手鳍。
- 蓝色脚套和黑鞋。
- 人体穿戴尺度：外壳只比真人肩宽和躯干略宽，头部和躯干不能放大成圆顶吉祥物或桶状气球。

左侧 / 右侧：

- 只能有 1 个黑色圆眼。
- 必须有 5 条黑色弧形鳃线。
- 橙色圆形鼓风阀必须在实际所在侧的侧腰，方向和高度正确。
- 左右侧鳍、侧缝、鞋底、侧面厚度和不对称关系必须稳定。

背面：

- 纯蓝背部。
- 中轴背部竖缝。
- 中轴尾鳍。
- 背部轮廓不能膨胀成无结构圆柱。

本地辅助角度 / 局部证据：

- 布料褶皱。
- 接缝张力。
- 阀门网格。
- 脸窗透明反光。
- 拉链和边缘缝线。

## 工程实现约束

前端：

- 上传区必须是四张平行卡片。
- 正面、左侧、右侧、背面四张核心图没齐时，“素材上传完成”禁用。
- 用户界面不提供细节图上传入口；长期产品的辅助角度由本地预设自动附带。
- 图片生成和文字提示词生成 API 由后端固定配置，用户界面不能要求用户填写图片/文字 API Key 或 base URL。
- 视频生成 API 可以保留界面控制，因为视频服务可能独立切换。
- 换任意产品图、换产品类型、换场景或换比例后，旧首帧、视频、质检状态必须作废。
- 用户看不到“定款”步骤。
- 用户看不到首帧 URL、参考图 URL 等技术字段。
- 图片预览必须 `object-fit: contain`，不能裁切产品。

后端：

- 图片生成和文字提示词生成统一使用后端 `IMAGE_TEXT_BASE_URL` / `IMAGE_TEXT_API_KEY`，不得把这组凭证传到浏览器表单。
- `/api/first-frame` 必须校验 `image_urls` 正好 4 张核心图。
- `/api/first-frame` 发送给多模态图像模型前，必须给四张核心图标注固定视图角色。
- `/api/first-frame` 可以接收本地预设 `support_image_urls`，但只能作为同款产品局部证据补充。
- 不再接受 `foreground_source_url` 作为首帧输入。
- prompt 必须明确“四视图是拓扑地图，不是拼贴要求”。
- prompt 必须明确“不要平均四张图生成新产品”。
- prompt 必须明确“不要把物理不可见的侧面/背面细节强行挪进当前视角”。
- 视频 prompt 必须继承已批准首帧和四视图产品锁。

验证：

- 每次改工作流必须运行 `npm run test:baseline`。
- 必须浏览器检查四步流程、五个本地产品预设、四张核心平行上传卡、无细节图上传区、禁用状态、隐藏定款步骤、隐藏 URL 字段、无横向溢出。
- 必须检查后端少于 4 张图会被拒绝，4 张图能通过四视图校验后进入 API Key 校验。
